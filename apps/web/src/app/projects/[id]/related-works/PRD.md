# Related Works (선행연구 조사) PRD

## 1. 개요

### 목적

연구자가 주제별 선행연구를 체계적으로 조사하고, 관련 문헌(연구과제보고서, 논문, 특허, 기사 등)을
검색·정리·저장할 수 있는 페이지.

### 사용자 페르소나

- **주 사용자**: R&D 연구자, 과제 기획자
- **Pain Points**:
  - 여러 플랫폼(NTIS, RISS, KIPRIS 등)을 일일이 방문하여 검색
  - 검색 결과를 체계적으로 정리하기 어려움
  - 과거 검색 히스토리 추적 불가
  - 중요 문헌을 북마크/메모와 함께 저장하기 번거로움

### 핵심 가치 제안

1. **통합 검색**: 연구과제보고서 17만건 + 논문/특허/기사/블로그 통합 검색
2. **히스토리 관리**: 주제별 검색 이력 자동 저장, 언제든 재조회
3. **스마트 갈무리**: 개별/일괄 북마크, 메모 첨부, 워크스페이스 내 저장
4. **연구자 친화 UX**: 카드/테이블 뷰, 필터/정렬, 빠른 미리보기

## 2. 기능 요구사항

### 2.1 검색 히스토리 (History List)

#### 2.1.1 히스토리 생성 방식

- **자동 생성**: 주제선택 페이지에서 주제 확정 시 자동으로 선행연구 검색 트리거 → 히스토리 생성
- **수동 생성**: Related Works 페이지 내 "새 검색" 버튼 → 키워드 입력 → 검색 실행 → 히스토리 추가
- **재검색**: 기존 히스토리 카드의 "재검색" 버튼 → 동일 키워드로 재검색 → 새 히스토리 생성 (날짜
  구분)

#### 2.1.2 히스토리 카드 구성

- **표시 정보**:
  - 검색일시 (예: `2025-02-14 14:30`)
  - 주제명/키워드 (예: `건설 현장 AI 안전 관리`)
  - 결과 개수 (예: `연구과제 48건 · 논문 127건 · 특허 32건`)
  - 검색 소스 (예: Badge로 `자동`, `수동`)
- **액션**:
  - 카드 클릭 → 결과 상세 페이지로 이동
  - "재검색" 버튼 → 동일 키워드로 새 검색
  - "삭제" 버튼 → 히스토리 삭제 (confirm 다이얼로그)

#### 2.1.3 히스토리 목록 레이아웃

- **기본 뷰**: 세로 스크롤 카드 리스트 (최신순)
- **정렬/필터**:
  - 정렬: 최신순 / 오래된순 / 결과 많은순
  - 필터: 검색 소스 (자동/수동), 날짜 범위
- **빈 상태**: 히스토리 없을 때 "주제선택 페이지에서 주제를 확정하거나, 새 검색을 시작하세요"
  메시지 + CTA 버튼

---

### 2.2 검색 결과 상세 (Result Detail)

#### 2.2.1 데이터 소스 및 우선순위

- **주 데이터**: 연구과제보고서 (17만건 DB) — 기본 탭
- **부가 데이터**: 논문, 특허, 신문기사, 블로그 — 참고용 탭
- **Mock 데이터 구조**:
  ```typescript
  {
    id: string
    type: "report" | "paper" | "patent" | "article" | "blog"
    title: string
    authors: string[]
    publishedAt: string
    organization?: string // 연구과제/논문의 수행기관
    abstract: string
    keywords: string[]
    url?: string // 원문 링크
    similarity: number // 0-100, 주제 유사도
  }
  ```

#### 2.2.2 탭 구조

- **Tabs**: `연구과제` (기본), `논문`, `특허`, `기사·블로그`
- 각 탭마다 독립적인 결과 리스트
- 탭 레이블에 결과 개수 표시 (예: `연구과제 (48)`)

#### 2.2.3 결과 리스트 뷰

##### 카드 뷰 (기본)

- **카드 구성**:
  - **Header**:
    - 타입 Badge (예: `연구과제`, `논문`)
    - 유사도 Badge (예: `93%`)
    - 체크박스 (좌상단)
    - 갈무리 버튼 (우상단, 북마크 아이콘)
  - **Body**:
    - 제목 (1줄, 텍스트 넘치면 ellipsis + tooltip)
    - 저자/기관 (1줄, 세미콜론 구분)
    - 발행일 (예: `2024-03-15`)
    - 키워드 (Badges, 최대 5개)
    - 초록 (3줄 미리보기, "더보기" 버튼)
  - **Footer**:
    - "상세보기" 버튼 → 우측 "미리보기" 패널 열림
    - "원문 보기" 링크 (외부 링크)

##### 테이블 뷰 (옵션)

- **컬럼**: 체크박스 | 제목 | 저자/기관 | 발행일 | 유사도 | 갈무리 | 액션
- 행 클릭 시 "미리보기" 패널 열림

#### 2.2.4 필터 & 정렬

- **필터**:
  - 발행연도 (슬라이더, 2010~현재)
  - 유사도 (슬라이더, 50~100%)
  - 키워드 (멀티셀렉트, 자동 추출된 키워드 목록)
- **정렬**:
  - 유사도 높은순 (기본)
  - 최신순
  - 오래된순

#### 2.2.5 일괄 액션

- **체크박스 선택**: 다중 선택 가능
- **상단 액션 바**:
  - "전체 선택" / "선택 해제"
  - "선택 항목 갈무리 ({n}건)" 버튼
  - → 클릭 시 선택된 항목들을 우측 "갈무리" 패널에 추가
  - → Toast 알림: "{n}건이 갈무리되었습니다"

### 2.3 갈무리 (Bookmark/Clip)

#### 2.3.1 갈무리 저장 위치

- **현재**: 클라이언트 메모리 (추후 DB 연동 예정)
- **미래**: 프로젝트별 DB 테이블 (`bookmarks` 테이블, `projectId` + `itemId`)

#### 2.3.2 갈무리 패널 (우측 Asidebar)

- **위치**: 우측 Asidebar의 "갈무리" 패널 (`AsidebarMenu` → `panelType: "갈무리"`)
- **구성**:
  - **리스트**: 갈무리된 항목 카드 (간략 버전)
    - 제목, 타입 Badge, 발행일
    - "메모 추가" 버튼 → 인라인 텍스트 입력
    - "제거" 버튼 (X 아이콘)
  - **액션**:
    - "전체 내보내기" 버튼 (CSV/JSON)
    - "메모와 함께 저장" 버튼 → 추후 DB 저장 기능

#### 2.3.3 갈무리 상태 관리

- **Zustand Store**: `useRelatedWorksStore`
  ```typescript
  {
    bookmarks: BookmarkItem[]
    addBookmark: (item: SearchResult) => void
    removeBookmark: (id: string) => void
    addMemo: (id: string, memo: string) => void
  }
  ```

### 2.4 미리보기 (Preview Panel)

#### 2.4.1 미리보기 패널 (우측 Asidebar)

- **위치**: 우측 Asidebar의 "미리보기" 패널 (`panelType: "미리보기"`)
- **구성**:
  - **Header**: 제목, 타입 Badge, 갈무리 버튼
  - **Body**:
    - 저자/기관
    - 발행일
    - 키워드 Badges
    - 초록 (전체 텍스트, 스크롤 가능)
  - **Footer**:
    - "원문 보기" 링크
    - "갈무리" 버튼

#### 2.4.2 미리보기 트리거

- 결과 카드의 "상세보기" 버튼 클릭 → 미리보기 패널 열림
- 테이블 뷰의 행 클릭 → 미리보기 패널 열림

## 3. 비기능 요구사항

### 3.1 성능

- 검색 결과 렌더링: 가상화 스크롤 (react-window) 사용, 100+건 결과도 부드럽게
- 히스토리 목록: 최대 100개까지 표시 (pagination)

- 키보드 네비게이션: 카드 간 이동 (`Tab`, `Arrow Keys`), 액션 실행 (`Enter`, `Space`)
- 스크린 리더: ARIA labels, 의미있는 alt text

### 3.3 반응형

- 현재 스코프 아님 (데스크톱 우선)

## 4. UI/UX 가이드라인

### 4.1 디자인 시스템

- **색상**: OKLCH 기반, 다크 모드 (zinc 팔레트)
- **타이포그래피**:
  - 제목: `text-lg font-semibold` (GowunBatang-Bold)
  - 본문: `text-sm` (Pretendard)
  - 캡션: `text-xs text-zinc-500`
- **간격**: Tailwind spacing scale (4, 6, 8, 12, 16, 24)

### 4.2 컴포넌트 패턴

- **Compound Component**: `RelatedWorks.Root`, `RelatedWorks.HistoryList`, `RelatedWorks.Detail`
- **Headless Hook**: `useRelatedWorks` (상태 관리 로직)
- **Radix UI**: Tabs, Checkbox, Accordion (shadcn/ui)

### 4.3 인터랙션

- **Hover**: 카드 hover 시 `bg-zinc-800` → `bg-zinc-750` 전환 (부드럽게)
- **선택**: 체크박스 선택 시 카드에 `ring-2 ring-primary` 효과
- **로딩**: 검색 중 Skeleton 로더 표시
- **빈 상태**: 일러스트 + 안내 문구 + CTA 버튼

### 4.4 피드백

- **Toast**: 갈무리 성공, 삭제 완료, 오류 메시지
- **Confirm**: 히스토리 삭제 시 "정말 삭제하시겠습니까?" 다이얼로그

## 7. 성공 지표

- **효율성**: 평균 검색-갈무리 시간 < 2분 (기존 10분 대비 80% 단축)
- **사용성**: 갈무리 기능 사용률 > 70%
- **만족도**: 연구자 UX 테스트 점수 > 8/10

## 8. 리스크 및 제약사항

### 8.1 기술 리스크

- **DB 연동 지연**: 백엔드 API 개발 전까지 Mock 데이터 사용 → 프론트엔드 개발 우선 진행
- **대용량 데이터 렌더링**: 17만건 중 검색 결과가 1000+건일 경우 → 가상화 스크롤 필수

### 8.2 UX 리스크

- **검색 결과 품질**: AI 기반 유사도 점수의 정확도가 낮으면 사용자 신뢰도 하락 → Phase 2에서 피드백
  수집 후 알고리즘 개선

### 8.3 제약사항

- **현재 스코프**: Mock 데이터 기반 UI/UX 구현만 (실제 검색 API 없음)
- **저장 위치**: 클라이언트 메모리 → 새로고침 시 갈무리 데이터 소실 (추후 DB 저장)
